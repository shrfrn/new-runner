## 1. What the Code Does

Exercise Runner is a web-based learning platform that allows students to work through programming exercises. It provides an interactive environment where users can:
- Browse and select exercises from a categorized sidebar
- Read exercise descriptions rendered from markdown files
- Run their JavaScript solution code directly in the browser
- Submit solutions to a test server for automated validation
- Copy exercise text as JavaScript comments for easy pasting into code editors
- View HTML-based documentation pages (Settings, About, Help)

The system maintains user preferences (like the last viewed exercise) in browser localStorage and supports browser navigation (back/forward buttons) with proper state management.

## 2. Main Features

### 2.1 Exercise Browsing & Navigation
- Hierarchical sidebar navigation with collapsible sections
- Support for both exercise (markdown) and documentation (HTML) content types
- URL-based deep linking using hash fragments
- Browser history integration with back/forward button support
- Active item highlighting and automatic section expansion

### 2.2 Exercise Display
- Markdown-to-HTML rendering using markdown-it library
- Clean, readable typography with syntax-highlighted code blocks
- Responsive layout with scrollable content areas

### 2.3 Code Execution
- Direct JavaScript execution of student solutions in the browser
- Dynamic script loading and reloading mechanism
- Console output for debugging

### 2.4 Testing & Submission
- Server-side testing via HTTP POST to `localhost:3000/api/test`
- FormData-based file upload
- Test results displayed in a new browser tab

### 2.5 Developer Convenience
- Copy-to-clipboard feature for exercise text (formatted as collapsible comments)
- Auto-run capability (optional setting)
- Persistent user settings via localStorage

### 2.6 Theme Support
- Light/dark mode using CSS `prefers-color-scheme` media queries
- Comprehensive CSS variable system for easy theming

## 4. High-Level Implementation

### 4.1 Application Initialization
On `DOMContentLoaded`, the app:
1. Loads user settings from localStorage
2. Flattens the hierarchical TOC structure for efficient lookups
3. Renders the sidebar navigation from the TOC data structure
4. Sets up event listeners for action buttons (Run, Test, Copy)
5. Registers `popstate` handler for browser navigation
6. Checks URL hash or settings to restore the last viewed item

### 4.2 Content Loading & Display
**Markdown Exercises:**
- Selected via sidebar click → triggers `loadMarkdownContent(id)`
- Fetches `.md` file from `/exercises/{folder}/{file}`
- Renders using markdown-it
- Updates URL hash and browser history
- Enables/disables buttons based on solution availability

**HTML Documents:**
- Selected via sidebar click → triggers `loadHtmlContent(folder, file)`
- Fetches `.html` file from `/html/{folder}/{file}`
- Injects content into main area
- Extracts and executes embedded scripts
- Disables exercise-related buttons

### 4.3 Code Execution Flow
1. User clicks "Run" button
2. `loadAndExecuteScript()` finds the corresponding solution file
3. Creates/replaces `<script>` element with solution file path
4. Browser loads and executes the JavaScript file
5. Console output appears in DevTools

### 4.4 Testing Flow
1. User clicks "Test" button
2. `sendScriptToServer()` fetches the solution file content
3. Creates a FormData object with file and exerciseId
4. POSTs to `http://localhost:3000/api/test`
5. Receives HTML report response
6. Opens report in new tab using Blob URL

### 4.5 State Management
A global `state` object tracks:
- `currentActiveItem` - DOM reference to active sidebar item
- `currentExercise` - ID of currently loaded exercise
- `flattenedToc` - Flattened array of all TOC items
- `isLoading` - Loading state flag
- `lastError` - Last error message
- `settings` - User configuration (lastEntryId, autoRun)

## 7. Main Code Flows & Scenarios

### 7.1 Initial Page Load
1. Browser loads `index.html`
2. `DOMContentLoaded` event fires
3. `init()` function executes:
   - Load settings from localStorage
   - Flatten TOC structure
   - Render sidebar HTML
   - Setup button listeners
   - Register popstate handler
4. Check for URL hash:
   - **If hash exists:** Find and click corresponding sidebar item
   - **If no hash:** Use `lastEntryId` from settings
   - **If neither:** Display default "Select an exercise" message
5. Application ready for user interaction

### 7.2 Exercise Selection
1. User clicks sidebar item
2. `onLoadItem(event, id)` handler executes
3. Prevent default link behavior
4. Extract item metadata (type, id, paths)
5. **For markdown exercise:**
   - Call `loadMarkdownContent(id)`
   - Fetch markdown file
   - Render using markdown-it
   - Inject HTML into content area
6. **For HTML document:**
   - Call `loadHtmlContent(folder, file)`
   - Fetch HTML file
   - Inject into content area
   - Execute embedded scripts
7. Update UI:
   - Set active class on sidebar item
   - Remove from previous item
   - Expand parent details section
8. Update browser:
   - Push history state
   - Update URL hash
9. Save to settings:
   - Store `lastEntryId` in localStorage
10. Update button states (enable/disable based on content type)

### 7.3 Running Solution Code
1. User clicks "Run" button
2. `loadAndExecuteScript()` executes:
   - Find exercise in flattened TOC
   - Extract solution file path
   - Check if script element exists
   - Remove old script if present
3. Create new script element:
   - Set ID to `exercise-script`
   - Set src to solution file path
   - Add error handler
4. Append to document:
   - Browser loads script
   - Script executes immediately
5. Output appears in browser console

### 7.4 Testing Solution
1. User clicks "Test" button
2. `sendScriptToServer()` executes:
   - Set loading state
   - Update button text to "Submitting..."
3. Fetch solution file content
4. Create File object from content
5. Build FormData:
   - Append file
   - Append exerciseId
6. POST to `http://localhost:3000/api/test`
7. Receive HTML response
8. Create Blob URL from HTML
9. Open in new tab with `window.open()`
10. Clean up Blob URL after 1 second
11. Update button text to "Submitted!"
12. Reset button after 2 seconds
13. **On error:** Log to console, reset button

### 7.5 Copy Exercise Text
1. User clicks "Copy" button
2. `copyExerciseAsComments()` executes:
3. Get rendered content from DOM (`innerText`)
4. Format as block comment:
   - Add `/*` at start
   - Prefix each line with `// `
   - Add `*/` at end
5. Copy to clipboard using Clipboard API
6. Update button text to "Copied!"
7. Reset button text after 2 seconds
8. **On error:** Show alert with error message

### 7.6 Browser Navigation (Back/Forward)
1. User clicks browser back/forward button
2. `popstate` event fires
3. Extract state data:
   - Check state type (html vs markdown)
   - Get item ID or folder/file info
4. **For HTML items:**
   - Find sidebar item by data attributes
   - Simulate click to load content
5. **For markdown items:**
   - Call `loadMarkdownContent(id, false)` (no loading indicator)
   - Call `updateActiveNavItem(id)` to sync UI
6. Content and UI restored to match URL

### 7.7 Auto-Run Feature
1. Exercise loads via `updateActiveNavItem()`
2. Check if item is a markdown exercise
3. Check if `settings.autoRun` is enabled
4. Check if exercise has a solution file
5. If all true:
   - Set 100ms timeout
   - Execute `loadAndExecuteScript()`
6. Script runs automatically after brief UI update delay

## 8. Code Design & Implementation Assessment

### Strengths

#### 8.1 Architectural Strengths
- **Clear separation of concerns:** UI, logic, services, and content are well-separated
- **Module-based architecture:** ES6 modules provide clean dependencies
- **Single source of truth:** TOC structure drives both navigation and functionality
- **Functional approach:** Consistent use of functions rather than classes
- **Event-driven:** Proper use of event listeners and handlers

#### 8.2 Code Quality
- **Readable code:** Functions are well-named and generally focused
- **No dependencies on frameworks:** Lightweight, fast-loading application
- **Browser API usage:** Proper use of modern browser features
- **State management:** Centralized state object prevents scattered variables
- **Error handling:** Try-catch blocks where needed, error state tracking

#### 8.3 User Experience
- **Persistent state:** User's position is saved and restored
- **Browser integration:** Back/forward buttons work correctly
- **Deep linking:** Shareable URLs for specific exercises
- **Visual feedback:** Button states change during operations
- **Performance:** Fast loading, minimal dependencies

#### 8.4 CSS Architecture
- **CSS Variables:** Comprehensive theme system with 36+ variables
- **Light/Dark mode:** Native browser preference support
- **Component-based:** Styles organized by component
- **Modern CSS:** Grid, custom properties, logical properties
- **Reset foundation:** Consistent baseline across browsers

### Weaknesses & Concerns

#### 9.1 Code Organization Issues

**Problem:** `exercise-runner.js` is 629 lines with many responsibilities
- Initialization, rendering, navigation, execution, testing, copying all in one file
- Violates Single Responsibility Principle
- Difficult to test individual features
- Hard to maintain as features grow

**Recommendation:** Split into multiple modules:
- `app.js` - Initialization and coordination
- `navigation.service.js` - Sidebar and routing logic
- `content.service.js` - Loading and rendering
- `execution.service.js` - Script running and testing

**Problem:** Global function attached to window object
- `window.onLoadItem` is an anti-pattern
- Makes code harder to test
- Pollutes global scope

**Recommendation:** Use event delegation instead:
```javascript
sidebar.addEventListener('click', (e) => {
  const item = e.target.closest('.sidebar-item')
  if (item) handleItemClick(item)
})
```

#### 9.2 Error Handling & Resilience

**Problem:** Inconsistent error handling
- Some errors are console-logged, some are alerted, some are logged to state
- No user-facing error UI component
- Silent failures in some cases

**Recommendation:**
- Implement consistent error handling strategy
- Add toast/notification component for errors
- Log all errors to state for debugging
- Provide helpful error messages to users

**Problem:** No retry logic for network failures
- Single fetch attempt with no retries
- No offline detection or handling
- No loading states for network operations

**Recommendation:**
- Add retry logic with exponential backoff
- Detect offline state and notify user
- Show loading indicators during network operations

#### 9.3 Security Concerns

**Problem:** Script execution without validation
- Direct execution of any JavaScript file
- No content security policy
- Potential for XSS if submissions folder is compromised

**Recommendation:**
- Implement Content Security Policy headers
- Consider sandboxing script execution (iframe, web workers)
- Validate file paths before loading

**Problem:** Hardcoded server URL
- `http://localhost:3000` is not configurable
- No environment-based configuration
- No HTTPS enforcement

**Recommendation:**
- Move to configuration file or environment variables
- Support multiple environments (dev, staging, prod)
- Enforce HTTPS in production

#### 9.4 State Management Issues

**Problem:** State mutations scattered throughout code
- Multiple functions directly mutate the global state object
- No state change notifications
- Difficult to track state changes
- Hard to implement features like undo/redo

**Recommendation:**
- Implement state management pattern (reducer-like)
- Single function for all state updates
- State change events for UI updates
- Consider immutable state updates

**Problem:** Settings and state are mixed
- Settings in localStorage, state in memory
- No clear boundary between persistent and ephemeral state

**Recommendation:**
- Clearly separate settings from runtime state
- Document what persists vs. what's ephemeral

#### 9.5 Browser Compatibility

**Problem:** No polyfills or fallbacks
- Uses modern APIs without feature detection
- Will fail on older browsers
- No graceful degradation

**Affected APIs:**
- ES6 modules
- Fetch API
- Clipboard API
- CSS Grid
- CSS custom properties

**Recommendation:**
- Add feature detection
- Provide fallbacks for critical features
- Document minimum browser requirements
- Consider transpilation for wider support

#### 9.6 Testing Infrastructure

**Problem:** No automated tests
- No unit tests
- No integration tests
- No test framework
- Manual testing only

**Recommendation:**
- Add test framework (Vitest, Jest)
- Write unit tests for services
- Write integration tests for main flows
- Add end-to-end tests (Playwright, Cypress)

#### 9.7 Accessibility Issues

**Problem:** Limited accessibility features
- No ARIA labels on buttons
- No keyboard navigation for sidebar
- No focus management
- No screen reader support

**Recommendation:**
- Add ARIA attributes
- Implement keyboard shortcuts
- Manage focus after route changes
- Test with screen readers

#### 9.8 Performance Concerns

**Problem:** TOC flattening on every page load
- Same data structure processed repeatedly
- Could be pre-computed or cached

**Recommendation:**
- Pre-flatten TOC during build
- Or cache flattened result

**Problem:** No code splitting
- All JavaScript loads upfront
- markdown-it library is 300KB+
- No lazy loading

**Recommendation:**
- Implement dynamic imports for large libraries
- Load markdown-it only when needed
- Split code into smaller chunks

**Problem:** No virtual scrolling for sidebar
- All exercises rendered even if not visible
- Could be slow with hundreds of exercises

**Recommendation:**
- Implement virtual scrolling for large lists
- Or progressive disclosure (load sections on demand)

#### 9.9 Developer Experience

**Problem:** No build process
- No minification
- No bundling
- No hot reload during development
- No linting or formatting enforcement

**Recommendation:**
- Add Vite or similar build tool
- Implement ESLint and Prettier
- Add pre-commit hooks
- Enable hot module replacement

**Problem:** No documentation generation
- No JSDoc comments
- No API documentation
- No inline code documentation

**Recommendation:**
- Add JSDoc comments to all functions
- Generate documentation site
- Add inline comments for complex logic

#### 9.10 Code Style Inconsistencies

**Problem:** Inconsistent string quotes
- Mix of single and double quotes
- No enforced style guide

**Problem:** Inconsistent function declarations
- Mix of function declarations and arrow functions
- No clear pattern for when to use which

**Problem:** Semicolon usage
- Comment in submission says "reviewed ✔️" but uses semicolons
- User rules say no semicolons, but config files have them

**Recommendation:**
- Enforce consistent code style with Prettier
- Document coding conventions
- Use ESLint to catch inconsistencies
