# Internal Routing System Overview
*Created: November 8, 2025*

## What the Routing System Does

The routing system implements client-side, hash-based navigation that allows users to:
- Navigate between exercises and content pages without full page reloads
- Use browser back/forward buttons to move through navigation history
- Share deep links to specific exercises via URL hashes
- Return to their last visited exercise on subsequent visits

## Main Features

1. **Hash-based Routing**: Uses URL fragments (`#id`) to represent content routes
2. **History API Integration**: Leverages `pushState` and `popstate` for browser navigation
3. **Deep Linking**: Supports direct navigation to specific content via URL hash
4. **State Persistence**: Remembers the last visited exercise across browser sessions
5. **Active State Management**: Updates UI to reflect current route
6. **Sidebar Integration**: Synchronizes navigation with sidebar interactions

## Tech Stack

- **Vanilla JavaScript** (ES6 modules)
- **Browser History API** (`history.pushState`, `popstate` event)
- **Browser localStorage** (via config service for persistence)
- **Hash-based URLs** (`window.location.hash`)

## How Routing is Implemented

### High-Level Flow

1. **Initialization**: On page load, the app determines the initial route from either:
   - URL hash (if present)
   - Last visited exercise (from localStorage)
   - Defaults to showing welcome message

2. **User Navigation**: When a user clicks a sidebar item:
   - Prevents default anchor behavior
   - Loads the associated content (markdown or HTML)
   - Updates URL hash via History API
   - Updates sidebar active state
   - Saves current route to localStorage

3. **Browser Navigation**: When user clicks back/forward:
   - `popstate` event fires
   - App reads state and navigates to that content
   - UI updates to match

### Content Type Handling

Routes map to two content types:

- **`ex-markdown`**: Programming exercises (markdown files)
  - Enables Run/Test/Copy buttons
  - Sets current exercise in app state
  
- **`html`**: Static pages (sign-in, settings, about, help)
  - Disables exercise-specific buttons
  - Special handling for auth page initialization

## Codebase Structure

The routing system is distributed across several focused modules:

### Core Files

```
src/js/
├── router.js              # Core routing logic
├── exercise-runner.js     # App entry point & routing setup
├── sidebar.js            # Navigation UI & active state
├── content-loader.js     # Route-based content loading
└── services/
    └── config.service.js # Route persistence

src/content/
└── toc.js                # Route definitions (table of contents)
```

## File Responsibilities

### `/src/js/router.js` (42 lines)
**Core routing module** - Exports three pure functions:

- `setupPopstateListener(onNavigation)`: Registers browser back/forward handler
- `handleInitialRoute(settings, onSidebarItemClick)`: Determines initial route on page load
- `pushHistoryState(id)`: Updates URL and browser history when navigating

**Key Logic**:
- Checks URL hash first, then falls back to `lastEntryId` from settings
- Triggers appropriate navigation by simulating sidebar clicks

### `/src/js/exercise-runner.js` (177 lines)
**Main application entry point** - Integrates all routing functionality:

- Initializes app state on `DOMContentLoaded`
- Calls `setupRouting()` which wires up router with content loading
- Defines `onLoadItem(ev, id)` as the global click handler for sidebar items
- Manages navigation through `handleItem(id)` which:
  - Loads content via `loadItem(id)`
  - Updates history via `pushHistoryState(id)`
  - Persists route via config service

**Interfaces with Router**: Lines 16, 124-126

### `/src/js/sidebar.js` (74 lines)
**Navigation UI generator and active state manager**:

- `buildSidebar(items)`: Recursively generates HTML for navigation links
- `renderSidebar(items)`: Injects sidebar HTML into DOM
- `updateActiveSidebarItem(id, currentActiveItem)`: Updates active state for routing

**Key Detail**: Each sidebar link includes:
- `data-item-id`: The route identifier
- `data-type`: Content type (ex-markdown or html)
- `onclick`: Inline handler calling global `onLoadItem`
- `href`: Hash link for accessibility

**Interfaces with Router**: Line 34 (generates links with route data)

### `/src/js/content-loader.js` (89 lines)
**Content fetching and rendering module**:

- `loadMarkdownContent(id, flattenedToc)`: Fetches and renders markdown exercises
- `loadHtmlContent(folder, file)`: Fetches and injects HTML pages

**Interfaces with Router**: Routes determine which loader function gets called and with what parameters (derived from TOC data)

### `/src/content/toc.js` (953 lines)
**Route definitions and content structure**:

Exports a configuration object defining all available routes:
```javascript
{
  sidebar: [
    {
      label: 'Exercise Runner',
      items: [
        {
          label: 'Sign In',
          type: 'html',
          id: 'auth',
          content: { folder: '.', files: ['sign-in.html'] }
        },
        // ... more items
      ]
    }
  ]
}
```

**Each route item defines**:
- `id`: Route identifier (used in URL hash)
- `type`: Content type (ex-markdown or html)
- `label`: Display text in sidebar
- `content`: File path configuration
- `solution` (optional): Associated solution files for exercises

**Interfaces with Router**: Provides the data structure that drives routing behavior

### `/src/js/services/config.service.js` (39 lines)
**Configuration persistence**:

- `loadConfig()`: Retrieves settings including `lastEntryId`
- `saveConfig(config)`: Persists settings to localStorage

**Interfaces with Router**: 
- Provides `lastEntryId` for initial route determination
- Receives updates when user navigates to save current route

## Main Code Flows

### Flow 1: Initial Page Load

```
User opens app
  ↓
DOMContentLoaded fires
  ↓
init() in exercise-runner.js
  ↓
setupRouting()
  ├─→ setupPopstateListener(loadItem)        # Register back/forward handler
  └─→ handleInitialRoute(settings, loadItem) # Determine initial route
        ↓
      Check window.location.hash
        ↓
      YES → Find matching sidebar item → Click it
      NO  → Check settings.lastEntryId → Find matching sidebar item → Click it
```

### Flow 2: User Clicks Sidebar Item

```
User clicks sidebar link
  ↓
onclick="onLoadItem(event, id)" fires
  ↓
Prevent default anchor behavior
  ↓
handleItem(id)
  ├─→ loadItem(id)
  │     ├─→ Find item in flattenedToc
  │     ├─→ Load content (markdown or HTML)
  │     ├─→ updateActiveSidebarItem(id)
  │     └─→ Update app state (currentExercise, currentActiveItem)
  │
  ├─→ pushHistoryState(id)
  │     └─→ history.pushState({ id }, '', `#${id}`)
  │
  └─→ Save to config
        └─→ settings.lastEntryId = id
```

### Flow 3: Browser Back/Forward Button

```
User clicks back/forward
  ↓
popstate event fires
  ↓
setupPopstateListener callback fires
  ↓
Check event.state?.id exists
  ↓
YES → loadItem(event.state.id)
        ├─→ Load content for that id
        ├─→ Update sidebar active state
        └─→ Update app state

NO  → Ignore (external navigation)
```

### Flow 4: Direct URL Access with Hash

```
User opens URL with hash (e.g., #42)
  ↓
handleInitialRoute() reads hash
  ↓
window.location.hash.substring(1) → "42"
  ↓
Find sidebar item with data-item-id="42"
  ↓
Programmatically click that sidebar item
  ↓
Triggers standard navigation flow (Flow 2)
```

## Implementation Approach Assessment

### Strengths

1. **Simplicity**: Clean, understandable hash-based routing without external dependencies
2. **Separation of Concerns**: Routing logic isolated in dedicated module
3. **Progressive Enhancement**: Links work as standard anchors, enhanced with JavaScript
4. **State Persistence**: localStorage integration for session continuity
5. **Browser Integration**: Full support for back/forward navigation
6. **Deep Linking**: Shareable URLs for specific content

### Design Patterns

- **Event-Driven Architecture**: Uses browser events (popstate, click) as triggers
- **Observer Pattern**: Router observes navigation events, notifies content loader
- **Configuration-Driven**: Route definitions centralized in TOC data structure
- **Functional Programming**: Router exports pure functions with no internal state
- **Inversion of Control**: Router accepts callbacks, doesn't know about content loading

### Trade-offs

**Chosen Approach**: Hash-based routing
- ✅ No server configuration required
- ✅ Works with static file hosting
- ✅ Simple to implement and debug
- ⚠️ Hash in URLs (aesthetic consideration)

**Alternative**: History API with path-based routing would require server-side rewrites

## Concerns and Potential Issues

### 1. Global Function Pollution
**Issue**: `window.onLoadItem` is attached to global scope (exercise-runner.js:88)

**Risk**: 
- Name collisions with other scripts
- Makes testing more difficult
- Violates module encapsulation

**Recommendation**: Use event delegation instead:
```javascript
// Instead of onclick="onLoadItem(event, id)"
// Attach single listener to sidebar container
document.querySelector('.sidebar-items').addEventListener('click', handleSidebarClick)
```

### 2. Inline Event Handlers
**Issue**: Sidebar links use inline `onclick` attributes (sidebar.js:34)

**Risk**:
- Violates content security policy if implemented
- Harder to test
- Mixed concerns (HTML generation contains behavior)

**Recommendation**: Generate data attributes only, use event delegation for behavior

### 3. Programmatic Click Simulation
**Issue**: Initial routing triggers navigation by calling `.click()` on DOM elements (router.js:17, 28)

**Risk**:
- Tight coupling between router and DOM structure
- Can't navigate if sidebar hasn't rendered yet
- Unusual control flow (router → DOM → event handler → router)

**Recommendation**: Call navigation handler directly:
```javascript
// Instead of: elSidebarItem.click()
onNavigation(id)
```

### 4. Missing Error Handling
**Issue**: No fallback for invalid route IDs

**Risk**: User follows broken link, sees empty content area

**Recommendation**: 
- Add 404-style handling for unknown IDs
- Redirect to default route or show error message

### 5. Race Conditions
**Issue**: No loading state management during navigation

**Risk**: 
- Rapid navigation could trigger multiple concurrent content loads
- Last load might not match current route

**Recommendation**: 
- Track current navigation in progress
- Cancel or ignore stale requests
- Disable navigation UI during load

### 6. TOC Data Structure Complexity
**Issue**: TOC mixes routing config, content paths, and UI structure

**Risk**:
- Hard to validate
- Easy to introduce inconsistencies
- No type safety

**Recommendation**: 
- Add runtime validation on app init
- Consider TypeScript or JSON schema validation
- Split routing config from content manifest

### 7. Hardcoded Route Structure
**Issue**: Router assumes numeric IDs can be parsed from strings (router.js:32-35)

**Risk**: 
- Breaks if ID scheme changes
- Special-case logic suggests missing abstraction

**Current Code**:
```javascript
const numberId = parseInt(lastEntryId, 10)
if (!isNaN(numberId) && onSidebarItemClick) {
    onSidebarItemClick(numberId)
}
```

**Recommendation**: Make ID handling consistent (all string or all number)

### 8. No URL Normalization
**Issue**: No canonicalization of route IDs

**Risk**: Same content accessible via different URLs (e.g., `#1`, `#01`, `# 1`)

**Recommendation**: Normalize hash before processing

### 9. Limited Accessibility
**Issue**: Navigation might not announce route changes to screen readers

**Recommendation**: 
- Update `document.title` on route change
- Use ARIA live regions for status updates
- Ensure focus management after navigation

### 10. No Loading States
**Issue**: Content loader shows "Loading..." but router doesn't manage global loading state

**Recommendation**: 
- Expose loading state for UI feedback
- Show loading indicator during navigation
- Disable navigation during pending loads

## Summary

The routing implementation is **clean, functional, and appropriate for a static exercise runner**. It successfully achieves client-side navigation with browser integration and state persistence using minimal code and no dependencies.

**Key strengths**: Simplicity, separation of concerns, and proper use of browser APIs.

**Main area for improvement**: Reducing coupling between router and DOM by eliminating programmatic clicks and inline event handlers in favor of event delegation.

The concerns listed are mostly **refinements rather than critical issues**. The system works well for its intended purpose, though production deployment should address the CSP and accessibility concerns.
