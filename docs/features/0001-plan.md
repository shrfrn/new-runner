# Sign In / Sign Up Page Plan

## Context
- Add a sign in/sign up page to the app so it appears under the Exercise Runner branch in the sidebar as the first option and is also accessible from a button in the top row of buttons next to Run, Test, etc.
- The page must include a two-tab form (sign in and sign up) that follows the same CSS used on the settings page to keep visual consistency.
- Sign in must include username (type=email for now) and password fields; sign up must also capture phone and fullname; all fields are required.
- Submitting either form will send a POST request to `/api/login` or `/api/signup` with form credentials, expect a mini user object in the response, store it in `localStorage`, and add a timestamped object to `sessionStorage` to distinguish between a new session and a browser refresh.
- After sign in, the header button must switch from “Sign In / Up” to a logout button that clears stored user/session data when clicked.

## Sidebar & Navigation Updates
- `src/content/toc.js`: Insert a new HTML entry for the sign in/sign up page before the existing `Settings` item within the Exercise Runner grouping so the new page becomes the first option.
- `src/js/sidebar.js`: No structural change needed, but confirm `buildSidebar` correctly renders the new item and wiring for `onLoadItem` works with the added entry.
- `src/js/exercise-runner.js`: Ensure `flattenToc` includes the new auth entry, update `loadItem` to load the new HTML page, and extend `handleItem` or related routing logic so clicking the header button triggers the same loading via the `onLoadItem` pathway.

-## Header Button Integration
- `src/index.html`: Add a new button element in the header actions container, positioned alongside Run, Test, and Copy. The button text should start as “Sign In” and reuse existing button classes or create a variant consistent with the current styling. Ensure the button toggles its label/state to “Sign Out” after authentication.
- `src/js/ui.js`: Extend `setupButtons` to accept both an auth navigation handler and a logout handler, plus expose helpers to switch the button text/state between “Sign In” and “Sign Out” when a user logs in or logs out. Maintain accessibility by announcing state changes if needed.
- `src/js/exercise-runner.js`: In `setupEventHandlers`, register the new auth button handler that calls `handleItem` with the new auth page ID and updates history state, mirroring other navigation flows, and wire the logout handler to call the auth service’s logout logic and update UI state.

## Auth Page Content & Styling
- `src/content/html/sign-in.html` (new): Create the sign in/sign up HTML page with:
  - Tab controls that allow keyboard navigation (ARIA `role="tablist"`, `role="tab"`, `role="tabpanel"`).
  - Two forms that share consistent markup: sign in form (username/email, password) and sign up form (username/email, password, phone, fullname) with `required` attributes.
  - Submit buttons that invoke JavaScript event handlers rather than default navigation.
  - Error message region with `aria-live="polite"` for accessibility.
- Styling approach:
  - Remove the embedded CSS from `settings.html` and move all shared form rules into a new stylesheet `src/css/forms.css`, following the existing `@import` pattern used by the CSS entry point (`src/css/index.css` should import `forms.css`).
  - Update both `settings.html` and `auth.html` to reference the shared classes only, leaving no inline styling so future form pages can reuse the module.

## Client Logic & API Integration
- `src/js/services/auth.service.js`: Extend beyond `getLoggedinUser()` with helper functions:
  - `export async function login(credentials)` to POST to `/api/login`.
  - `export async function signup(details)` to POST to `/api/signup`.
  - `export function saveLoggedinUser(user)` to persist the response in `localStorage` under `loggedInUser`.
  - `export function markSessionStart()` to capture `{ timestamp: Date.now() }` (or similar) in `sessionStorage` under a new key (e.g. `sessionMeta`). Include error handling for fetch failures and JSON parsing.
  - `export function logout()` to remove all auth-related entries from both storage layers and return a status so the UI can reset immediately.
- `src/js/pages/auth-page.js` (new): After `loadHtmlContent` renders the auth page, this module should:
  - Initialize tab state with the sign in tab active.
  - On form submit, prevent default, validate that all required inputs are non-empty, call the relevant `auth.service` function, and, upon success, save the returned mini user object plus the session timestamp.
  - Show success or error feedback on the page without navigating away; consider keeping focus management accessible.
  - Notify callers when authentication succeeds so `exercise-runner.js` can update the header button to the logout state immediately.
- `src/js/content-loader.js`: Limit changes to exposing the rendered `.html-content` container (or a callback hook) so the auth page module can attach listeners without bloating the loader.
- Ensure all new JavaScript follows the project’s functional style guidelines (no classes, prefer small functions with early returns).

## State Persistence & Error Handling
- After a successful response, call `saveLoggedinUser` to store the mini user object exactly as received, then call `markSessionStart` so session metadata exists for refresh detection.
- When logging out, use the new `logout()` helper to clear both storage entries and toggle the header button back to the “Sign In / Up” state while clearing any page-specific feedback.
- Provide a graceful failure path: display API errors within the auth page, avoid throwing uncaught errors in the console, and allow resubmission after correcting issues.
- Consider future compatibility with other modules that may query the logged-in user by leaving the storage key and object schema consistent.

## Testing & Verification Notes
- Manually verify that:
  - The new sidebar item appears first under Exercise Runner and loads the auth page.
  - The new header button loads the same page and updates browser history.
  - Form validation prevents submission when required fields are empty.
  - Successful submissions persist data in `localStorage` (`loggedInUser`) and `sessionStorage` (timestamp object).
  - Styling matches the settings page and remains accessible (keyboard navigation, tab announcements, visible focus states).
- Consider adding mock or stub responses during development if backend endpoints are unavailable so the UI flow can be tested end-to-end.
